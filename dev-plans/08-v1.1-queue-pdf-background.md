# Plan 08: v1.1 — Reading Queue, PDF Support & Background Audio

**Estimated effort:** 5–6 days
**Depends on:** Plan 07 (v1.0 released)
**Phase:** v1.1 (PRD Phase 2)

## Objective

Add the three most-requested post-launch features: a reading queue for multi-article sessions, PDF text extraction, and background audio playback when the user navigates away. After this plan, users can queue up articles, listen to PDFs, and keep listening as they browse.

## Tasks

### 1. Reading queue (`/src/lib/queue.ts` + UI in popup)

- **Data model**:
  ```typescript
  interface QueueItem {
    id: string;
    url: string;
    title: string;
    favicon: string;
    addedAt: number;
    position?: number;       // last chunk index (for resume)
    percentComplete?: number;
  }
  ```
- **Storage**: `chrome.storage.local` key `readingQueue`
- **Operations**:
  - Add current page to queue (from popup button) — FR-36
  - View, reorder (drag-and-drop), remove items — FR-37
  - Persist across browser sessions — FR-38
  - Auto-advance: when current page finishes, open next queue item and start playback — FR-39
- **Queue panel UI** in popup: list of items with favicon + title + progress + remove button
- **Service worker logic**: on playback end, check queue → `chrome.tabs.update()` to navigate → trigger extraction + playback

### 2. PDF support (`/src/content/pdf-extractor.ts`)

- Detect Chrome's built-in PDF viewer (`<embed type="application/pdf">` or URL ending in `.pdf`)
- Integrate `pdf.js` (Mozilla's PDF rendering library):
  - Load the PDF document from the current page URL
  - Extract text content page-by-page via `page.getTextContent()`
  - Concatenate into a single text string, preserving paragraph breaks
- Feed extracted text into the existing chunker pipeline
- Handle edge cases:
  - Scanned PDFs (no text layer) → show user-friendly error
  - Large PDFs (100+ pages) → progressive extraction, don't block UI
  - Password-protected PDFs → error message

### 3. Background audio (`/src/offscreen/`)

- Use the Offscreen Document API to keep audio playing when:
  - User switches to a different tab
  - User minimizes the browser
- The offscreen document already exists from Plan 03; ensure it survives tab navigation:
  - `chrome.offscreen.createDocument({ reasons: ['AUDIO_PLAYBACK'], justification: '...' })`
  - The offscreen doc lives independently of any tab
- Handle the case where Chrome garbage-collects the offscreen doc:
  - Service worker re-creates it and resumes from last position

### 4. Per-site voice preferences (`/src/lib/site-preferences.ts`)

- Allow users to assign a specific provider + voice to a domain
  - e.g., "Always use ElevenLabs / Rachel on medium.com"
- Storage: `{ [domain]: { providerId, voiceId } }` in `chrome.storage.local`
- UI: add a "Set voice for this site" option in the toolbar's expanded state
- Settings page: manage per-site rules in the Advanced section

### 5. Reading history (P2 — stretch goal)

- Track last 50 pages read: URL, title, position, % complete — FR-40
- Resume from last position when revisiting a URL — FR-41
- History panel accessible from popup
- Clear history button

## Exit Criteria

- [ ] Users can add pages to a reading queue from the popup
- [ ] Queue items can be reordered and removed
- [ ] Queue persists across browser restarts
- [ ] Auto-advance plays the next queue item when current finishes
- [ ] PDF files are extracted and read aloud correctly
- [ ] Scanned PDFs (no text) show an appropriate error
- [ ] Audio continues playing when user switches tabs
- [ ] Audio continues when browser is minimized
- [ ] Per-site voice preferences work and persist
- [ ] (Stretch) Reading history tracks last 50 pages with resume support

## Deliverables

- Reading queue system (data layer + popup UI + auto-advance)
- PDF text extraction via pdf.js
- Background audio via Offscreen Document API
- Per-site voice/provider preferences
- (Stretch) Reading history with resume
